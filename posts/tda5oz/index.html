别再让 Claude 直接碰你的宿主机了：我的沙盒化远程开发方案

|浏览量:

前言

Claude Code 很好用，但我越来越不想让它直接跑在宿主机上。

原因很简单。一个能读文件、改代码、跑命令、装依赖的 Agent，如果直接工作在日常开发机上，边界其实很弱。哪怕没有恶意，单是一次误操作、一次错误安装、一次不够谨慎的文件修改，都足够把本地环境弄脏。更现实的问题是，本地 CLI 只能在“我坐在电脑前”的时候工作，关掉终端、断开网络、离开工位，这个协作关系就中断了。

所以我最后做的，不是“优化 Claude Code 的使用体验”，而是把它的运行方式重做了一遍：宿主机负责隔离，Linux 沙盒负责执行，沙盒内再补齐远程控制、持久化、记忆和自动化。

这样 Claude Code 就不再只是一个本地终端工具，而变成了一个可远程操控、可长期运行、可快速恢复的 AI Agent。

此外这样还有一个好处是：Claude Code 在沙盒内的内容都是受控的，你可以将沙盒内的语言、时区都改成美国，再对沙盒网桥设置代理，将自己伪装成一个美国用户。

设计目标

这套方案的目标很明确：

  • 不让 Claude 直接污染宿主机环境
  • 保持正常开发体验，而不是为了隔离牺牲可用性
  • 支持长期运行，而不是一次性会话
  • 容器重建后能快速恢复
  • 支持远程触发、定时执行和跨会话记忆

我的威胁模型也很明确：这是个人开发机方案，不是企业级零信任安全体系。重点是“宿主机别被搞乱”，而不是把所有网络和权限都收紧到极致。因此我没有做严格的出网白名单，也没有做极细颗粒度的权限拆分，而是选择一个更实用的平衡点。

总体架构

整套方案可以拆成两部分。

第一部分是宿主机侧沙盒。宿主机只负责这些事情：用 Docker 提供一个固定的 Linux 容器、把项目目录统一挂载进去、提供代理转发、提供启动入口、把需要长期保存的用户态数据放进 volume。

第二部分是沙盒内 Agent 化。把 Claude 装进容器，只解决了“隔离”；它还远远不是一个真正可长期协作的 Agent。要做到这一点，容器内还需要补齐四层能力：

  1. 远程控制层：让 Claude 能通过 IM 接受消息
  2. 持久化层：让配置和行为约定可恢复
  3. 记忆层：让 Claude 能跨会话保留长期信息
  4. 自动化层：让重复任务能定时执行

这四层补齐以后，Claude 才从“本地 CLI”变成“远程 Agent”。

宿主机侧：我解决的是隔离和可重建

宿主机这一层，最核心的设计有四个。

1. 所有项目统一挂载

我强制所有项目都放在 ~/cc_mnt 下，启动函数会校验路径，不在这个根目录里的项目直接拒绝。容器内路径统一映射到 /workspace/<相对路径>，这样宿主机和容器里的路径语义是一致的。

这件事的作用，不只是方便管理，更是在给 Agent 画边界。

2. 容器长期存在，不使用 --rm

如果使用一次性容器，退出后可写层就没了，很多运行时状态也会丢。对普通命令行工具来说无所谓，但对一个长期协作的 Agent 来说，这很伤。

所以我的容器是长期存在的：运行中就直接进入，停止了就启动后进入，只有第一次或者显式删除后才重新创建。这样做的好处是状态稳定、进入快，而且“重建容器”变成了一个显式操作，而不是每次退出都隐式发生。

3. /home/node 放进 named volume

真正需要保留的东西，不该写死在镜像层，而应该放进 volume。

比如这些内容都放在 /home/node 下持久化：

  • Claude 的登录状态和历史
  • Codex 的认证信息
  • cc-connect 配置和会话数据
  • shell 配置
  • 自定义脚本
  • 记忆文件

这样容器即使被删掉，只要 volume 还在，核心状态就还在。

4. 镜像层只放“应该长期存在的工具”

有一个很容易踩坑的点：如果某个二进制是运行时手工装进去的，而它又落在镜像层或容器可写层，那容器重建后它就消失了。

所以像 codex、cc-connect 这种属于“沙盒基础设施”的工具，我直接写进 Dockerfile，而不是等容器起来后手工安装。这样重建容器时，工具本身就是确定存在的，不依赖手工补壳。

这里要注意镜像的启动脚本要带 --init 参数来启动一个 tini 进程作为 PID1，不然 claudecode 很容易留下孤儿进程而无法回收！

沙盒内：从 CLI 到远程 Agent 的四层补强

真正有意思的部分在容器内。一个新装好的 Claude Code，本质上只是本地 CLI。要把它变成远程 Agent，需要补四层东西。

第一层：远程控制

我用 cc-connect 把 Claude Code 接到了 Telegram。

这样做的意义不是“我可以在 Telegram 里和 Claude 聊天”，而是：Claude 的主执行环境始终留在沙盒内。 我在手机上发一条消息，沙盒里的 cc-connect 会把它交给本地 claude 进程，再把结果返回给 Telegram。

一旦接通这一层，交互模式就变了：

  • 不需要 SSH 进机器才能用
  • 不需要守着终端
  • 关掉手机或聊天窗口后，Claude 仍然活着
  • 过一段时间回来，它还能继续前一轮上下文

这才是真正意义上的远程开发，而不是“远程打开终端”。

这里有个关键配置是 bypassPermissions。远程场景下必须开，不然 Claude 每次调用 Bash 或 Edit 都会卡在权限确认。但这件事的前提是：沙盒边界必须够紧。 如果沙盒外资源仍然可访问，或者密钥仍然是高价值主密钥，那这相当危险。

第二层：持久化

把 Agent 做成长期协作者，核心不是“怎么安装”，而是“怎么恢复”。

Claude 的很多用户级配置都放在 ~/.claude 下，所以我直接把 ~/.claude 做成一个 git 仓库来管理，只提交稳定配置，不提交运行时数据。这样容器重建以后，重新 clone 一次就能恢复大部分行为约定。

我会跟踪这些内容：

  • settings.json
  • 自定义 skills
  • statusline 脚本
  • watchdog 脚本
  • 项目级记忆文件
  • 全局行为约定文件

而这些运行时数据全部排除：

  • sessions
  • cache
  • telemetry
  • history
  • credentials
  • 任务执行痕迹

这层设计的价值非常直接：环境可以被丢弃，但配置不需要重新发明。

第三层：记忆

会话历史不是记忆，记忆也不该是历史会话的全文备份。

我在 ~/.claude/projects/<workspace>/memory/ 下维护项目级 memory，把真正应该长期记住的信息提炼成 markdown 文件。比如这些内容就很适合写进 memory：

  • 不同主题必须拆成不同分支和 PR
  • 并入主干优先使用 rebase 而不是 merge
  • cc-connect 不支持发图片，要直接走 Telegram Bot API

而这些内容则不适合进 memory：

  • 代码结构
  • 文件路径
  • git log 能查到的信息
  • 会话内临时上下文
  • 已经写在 AGENTS.md 或项目文档里的规则

memory 的真正价值，是把“磨合成本”从无限重复，降低成有限次数。

第四层：自动化

如果一个任务每天都要做，那它就不该永远靠手动触发。

我直接用了 cc-connect 自带的 cron，把定时 prompt 注入到固定会话里。这样触发的是一轮正常对话，而不是一个孤零零的 shell 命令。它可以照常读取 memory、调用 skill、使用已有工具链。

我现在主要自动跑两类任务：

  • 每日生成日报
  • 每日备份 ~/.claude

为什么不用系统 crontab？因为系统 cron 更适合跑命令，而不是驱动一个带上下文、带工具调用、带状态记忆的 Agent。既然任务本来就该通过对话完成，就应该让它从 Agent 自己的入口被调起。

多 Agent 共存：Claude 和 Codex 共用一套行为约定

我在沙盒里同时装了 Claude Code 和 Codex。两者都需要一份全局行为约定，比如：

  • 全程中文沟通
  • 修改前先理解现有实现
  • 只动必要部分
  • 优先可验证、可回滚的改动
  • 统一 commit message 规范
  • 未验证不算完成

为了避免维护两份，我把 ~/.claude/CLAUDE.md 直接链接到 ~/.codex/AGENTS.md，把这份文件当作单一事实源。

这样做的好处很直接：无论是 Claude 还是 Codex，启动后遵守的都是同一套规则。全局行为统一，仓库内再用项目自己的 AGENTS.md 做局部覆盖，结构就很清楚。

这套方案真正解决了什么

如果只看某一项改动，这些设计都不算惊人：用容器隔离、用 IM 做远程控制、用 git 管理配置、用 memory 管理长期信息、用 cron 自动触发任务。

但它们叠在一起以后，效果会发生质变。Claude 不再只是“一个我打开就用一下的终端工具”，而是一个具备这些特征的工作环境：

  • 可远程操控
  • 可长期存活
  • 可跨会话保留经验
  • 可自动执行重复任务
  • 可快速恢复

这就是我真正想要的结果：不是让 Claude 更像一个会聊天的 CLI，而是让它更像一个可长期协作的开发 Agent。

我刻意没有做的事

这套方案不是企业级零信任系统，也不是追求极限安全的方案。我没有做这些事：

  • 不强制所有出网白名单
  • 不做极细粒度 sudo 限权
  • 不把容器做成一次性无状态环境
  • 不试图把宿主机凭据只读复用进容器

原因很简单：这些事会显著增加复杂度，而且未必符合个人开发者的实际收益。

我的目标是构建一个足够安全、足够稳定、足够容易恢复、并且真的能每天用的系统，而不是一个理论上最完美、实际上很难长期维护的系统。

结语

直接在宿主机上跑 Claude Code，当然能用。
但如果你真的想把它当成长期协作者，而不是一次性工具，那你迟早会遇到这些问题：

  • 它应该运行在哪里
  • 它挂了以后怎么恢复
  • 它怎么远程接入
  • 它怎么记住长期信息
  • 它怎么自动完成重复任务
  • 它怎么在隔离环境里持续工作

我这套方案，本质上就是对这些问题的一次工程化回答。

最后得到的结论也很明确：Claude Code 不该直接碰宿主机，也不该只活在你眼前的终端里。 更合理的形态，是把它放进一个边界清晰的沙盒里，再把这个沙盒养成一个可远程、可持久化、可自动化的 AI Agent。
