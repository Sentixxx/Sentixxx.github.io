代码里没写下来的那条规矩：AI 时代被放大的老 bug

更新于2026-04-20|思考
|浏览量:

本文部分内容采用 AI 辅助生成，核心素材与推演逻辑均提炼自排查该 Bug 时的真实 Session 对话文本。

越读越烂，返工一下 —— 2026.4.20

一个矛盾的回复

事情的起因源于 cc-connect 的一个 bug：同一分钟内，Telegram 群收到两条互相矛盾的回复

  • 一条说"会话已结束"
  • 另一条说“前一条请求仍在处理”。

显然这是矛盾的。

我很快梳理出了问题的链路，并提交了响应的 PR。

问题排查

第一层，agent 进程没死透。

agent/claudecode/session.go 的 readLoop 用 bufio.Scanner 读 stdout。Claude CLI 退出时，若有子孙进程继承了 stdout fd，管道不会 EOF，scanner 就一直阻塞。cs.ctx.Done() 触发时，defer 里虽然会关 channel，但那一路逻辑分支没有关 stdout，于是 readLoop 永远不返回。

PR 的修法是加一个 watchdog：ctx 取消或 cmd.Wait() 返回后，给 50ms grace period 让 scanner 把内核 pipe buffer 里最后几行 drain 出来，再强关 fd。

第二层，engine 有两份真相。
core/engine.go 维护一张 interactiveStates map，core/session.go 的 Session 结构体自己也有 busy 字段。
同一个会话的状态被两处分别记录，生命周期还错位—— cleanupInteractiveState 先删 map entry，再异步调 closeAgentSessionWithTimeout。

在这个异步的窗口里，新消息查 map 查不到，走"创建新会话"路径；Session 对象还活着，继续往同一个 chat 推流。两条回复由此产生。

但这还没完，让我好奇的是，究竟为什么会产生两份不同的 truth，为了探究这个问题，我让 agent 帮我梳理了一遍, 于是就有了第三层

第三层，四次 commit 累积出来的裂痕。

  • 3b097a8 初始化：Session 带 busy，合理。
  • ecd0e00 引入 interactiveStates map：为了 engine 层调度，合理。
  • f292073 给 cleanup 加 timeout，改成"先删 map 再异步 close"：为了避免 cleanup 卡主流程，合理。
  • 4089f78 handleMessage 开始读 map 判断忙态：为了队列化并发消息，合理。

每一步都正确。拼到一起就散架。

AI 的盲区

这种"单看每步都对，合起来散架"的情况过去就不是新鲜事，大型系统里屡见不鲜。但过去至少有两道护栏：作者本人多半读过项目历史，PR 合入前还有人类 reviewer 再看一遍。AI 开始批量提 PR 后，这两道都被绕过了。开源项目里几百条 “Generated by Claude Code” 的 PR 堆着没人合——这当然也算进步（很多 issue 过去根本没人动），但代价是reviewer 从门禁被压成瓶颈，而提 PR 的一方既不知道项目历史、也不会主动问。

这不是维护者水平问题——过去人审 PR 也审不出全部跨 commit 的语义漂移。但 AI 至少在三件事上把原来的裂缝撕大了：不主动追溯 git log，眼前 diff 就是它眼里的全部真相；不会问作者意图，默认代码已经说清楚了自己；写 PR 的速度远快于 reviewer 读懂它的速度。三件事叠起来，"局部正确"的增速第一次超过了"整体一致性"的维护速度。

那到底 AI 看不见什么？

interactiveStates 在代码里就是一个并发安全的 map，有 Get/Set/Delete，用锁保护。AI 读到的就是这些——而且都对。它看不见的是，当初写这个 map 的人脑子里其实压着一条没写进代码的规矩：map 里有这个 key，就说明会话在忙。这条规矩反过来说才是它真正在管的事——不允许出现两份独立的忙态记账。这种"不允许"的话谁都没法写进类型、写进接口，它只能留在作者脑子里，运气好的话偶尔漏到 commit message 里一两句。

更麻烦的是，现在写代码的常常不是人，是 AI 自己。作者那条没说出口的规矩，这下连脑子都没有了——它就挂在写这段代码时那个 AI session 的上下文里，session 一关，什么都不剩。人写的代码好歹还能翻 PR 讨论把意图找回来，AI 连这个可翻的东西都留不下。六周后下一个 AI 接手，面对的是一份彻底失忆的代码库，所有约束只能靠它当场从代码字面重新猜。

而且它还得跨着时间猜。f292073 那次把 cleanup 改成异步，站在当时那个 session 的角度看完全合理——它要解决的就是"cleanup 卡主流程"这个眼前的小问题。它没法知道六周前写 map 的人心里假设的是"delete 和 close 得一起做完"。作者的意图是随人走的东西，人在就在，人走了就散；代码不一样，它进了仓库就留下了，但它只记录你做了什么，不记录你为什么这么做。这就是所有"每步都对、合起来散架"类 bug 的根——这毛病不是 AI 独有，但 AI 恰好撞在枪口上：它最擅长读代码，而代码恰恰是这件事讲得最不清楚的地方。

能做点什么？

面对这种局面，常见的第一反应是"再回头把 Spec Driven Development 和 Test Driven Development 做到位"。这话不能说错，但它回避了问题的形状。

cc-connect 的四次 commit，每一次都完全可以配齐单测、过 CI、跑通集成测试——事实上也真跑通了，否则合不进 main。要让 TDD 拦下这个 bug，得有人先写一条测试，类似"cleanup 之后，Session 的 busy 状态不能和 map 里的 key 存在与否脱节"。这种测试不是不能写，问题是要写它，前提是先有人意识到这两份状态之间存在一条需要守护的关系。写第一版 map 的人(或者 AI)当时没把这条约束当不变量，只当它是个顺手写下来的实现细节；六周后改 cleanup 的人更没机会知道有这么一条东西存在。TDD 不能替你完成"识别出这里有条隐藏约束"这一步——它只能验证你已经意识到的东西。

而这一步，人偶尔能靠直觉兜住。一个在项目里泡了一段时间的老手，扫到 interactiveStates 的时候会短暂停一下："等等，这个跟 Session 的 busy 字段是不是在讲同一件事？"这个停顿不是 TDD 教的，是在一个 codebase 里待久了沉淀下来的隐性知识。它不保证每次都兜住，但它是在测试覆盖率之外多出来的一层托底。

AI 写代码的时候，这层托底直接没有。它读代码不缺细致，缺的是"这事儿怎么看着有点眼熟"的那种直觉。它会把 map 当 map，把 busy 当 busy，不会把两者连起来看。过去"测试写不到完备"的时候至少还有这种直觉兜底；AI 批量介入之后，连兜底那一层也被抽走了。

TDD 不顶事、人类直觉又被抽走之后，真正能补这一刀的其实是 Agent 框架自己。现在大多数框架只把自己当个"跑 LLM 的壳子"——用户发指令、LLM 调工具、工具返回结果。这个壳子管不了"AI 要改的这段代码，它的历史意图有没有被看到"。

框架层面有一件事值得硬做：把"修改前必须看历史"变成硬约束，而不是自觉。 现在 Claude Code 这类工具里，Agent 会不会去读 git blame、翻 PR 讨论，基本取决于它自己想不想做——而现实是大多数情况下它不想做，因为直接读当前代码就能把"眼前的问题"解决掉。可以在 harness 层面加一条规矩：每次要编辑一个函数之前，先强制拉这个函数最近的几次 commit message 和相关 PR 讨论塞进上下文，再决定怎么改。这事靠 prompt engineering 完成不了（Agent 总能绕过），必须是 hook 层强制的。回到 cc-connect，如果改 f292073 那个 session 在动手前被塞一段"六周前这个 map 是这么引入的，作者说要保证 delete 和 close 原子"，它多半不会把 cleanup 改成异步——至少会停下来问一句。

再往前一步：git log 和 PR 讨论只是意图的压缩包，失真率很高。AI 写代码的场景里，真正完整的"作者写这段代码时到底在想什么"其实可以原生保留——那次 session 的完整上下文、Agent 的思考过程、它读过哪些文件跑过哪些测试、当时 memory 和环境里有什么、考虑过哪些方案又否掉了、用户有没有追问过它怎么答——这些过去是锁在作者脑子里拿不出来的东西，现在全都以 session 形态完整存在，只是我们默认把它当一次性产物丢掉了。框架应该做的是把这些沉淀成和 commit 绑定的可查询资产：每次 AI 产出的 commit 除了 diff 和 message，额外挂一份 session 快照，以 commit SHA 为索引归档。下一个 session 改同一块代码时，框架把这份快照喂进上下文——不再靠 commit message 那一两句话去反推意图，而是直接把当初的"脑子"整份拿来读。以前说"意图跟着 session 一起散"，散是因为我们没把它接住；接住了，它就不散，至于如何管理如此庞大的上下文，这又是另一个问题了。

就算这么做，也有治不到的地方。人类作者写的代码没有 session 可复读，只能还靠 commit message 和 PR 讨论去反推；AI session 本身如果动手时就没意识到某条约束，那份快照里一样查不出来——盲点不会因为被归档而消失，只是不再被时间加剧。框架能做的只是把"局部正确合起来散架"的概率压低，压不到零。承认这一点，才不会把 Agent 框架当成下一颗银弹去卖。

第 1 步：状态 B（session.busy + TryLock）

Commit: 3b097a8 (2026-02-28 10:38) — init（仓库首提交）

从 day 1 就存在。core/session.go 里 Session 自带 mu sync.Mutex + busy bool，TryLock/Unlock 的语义就是「这个会话是否正在处理中」。

Go

第 2 步：状态 A（interactiveStates map）

Commit: ecd0e00 (2026-02-28 22:05) — feat(claudecode): refactor agent into persistent session model …（同一天晚些）

为了支持持久化 claudecode session + 权限状态，在 core/engine.go 里引入 interactiveStates map[string]*interactiveState 和 interactiveMu。从这一刻开始，「turn 在跑」有了两份独立的记账。

在这个 commit 里 cleanup 还是原子的（mu 锁住整个 Close + delete），所以两份状态的视图虽然分裂但在时间维度上没有窗口差：

Go

同期的 cmdStop 也只读 map（和今天一样）。这两份真相此时已经存在，但由于 cleanup 在 mu 下串行化，读者要么看到 state-still-present、要么看到 state-gone，没有「map 空但 agent 还在收尾」的中间态。

第 3 步：把「原子 cleanup」拆成「先 delete 后 async-close」

Commit: f292073 (2026-03-04) — feat: add timeout handling for agent session closure

这个 commit 改了 cleanupInteractiveState：先 delete 再释放 interactiveMu，然后把 Close 丢到新 goroutine 里用 10s timeout 等（后面涨到 130s）：

Go

这是两份真相第一次在时间维度分开。map 项消失、但 session lock 和 Close 仍在生效。原来被 interactiveMu 吸收掉的等待时间现在泄漏成了一个「状态已删但工作仍在」的窗口。

第 4 步：让 handleMessage 的 busy 分支开始读 map

Commit: 4089f78 (2026-03-16) — feat: queue messages when agent is busy instead of discarding them

这之前，handleMessage 走到 busy 分支是无条件回 MsgPreviousProcessing——和 cmdStop 一样只根据一个信号决定回复，两者不会矛盾。

这个 commit 引入了 queueMessageForBusySession：TryLock 失败后先去读 interactiveStates 决定「排队还是回 busy」。这就让 handleMessage 的回复策略变成了 session.busy + interactiveStates 联合判定，而 cmdStop 仍然只看后者。

Go

从这一刻起，同一份瞬态「map 空 + session lock 持有」下：

  • handleMessage 走 queue miss → MsgPreviousProcessing（因为 TryLock 失败且 queue 说没 state）
  • cmdStop 走 stopInteractiveSession miss → MsgNoExecution
