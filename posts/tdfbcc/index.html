Vibe Coding 时代的隐性代价：谁来为 AI 的“局部正确”买单？

更新于2026-04-13|思考
|浏览量:

如果你在其他平台看到这篇文章，这可能不是最终版本。为了获得更好的阅读体验（包含最新的评论讨论和勘误），欢迎移步原文

本文部分内容采用 AI 辅助生成，核心素材与推演逻辑均提炼自排查该 Bug 时的真实 Session 对话文本。

一个矛盾的回复

事情的起因源于 cc-connect 的一个 bug：同一分钟内，Telegram 群收到两条互相矛盾的回复

  • 一条说"会话已结束"
  • 另一条说“前一条请求仍在处理”。

显然这是矛盾的。

我很快梳理出了问题的链路，并提交了响应的 PR。

问题排查

第一层，agent 进程没死透。

agent/claudecode/session.go 的 readLoop 用 bufio.Scanner 读 stdout。Claude CLI 退出时，若有子孙进程继承了 stdout fd，管道不会 EOF，scanner 就一直阻塞。cs.ctx.Done() 触发时，defer 里虽然会关 channel，但那一路逻辑分支没有关 stdout，于是 readLoop 永远不返回。

PR 的修法是加一个 watchdog：ctx 取消或 cmd.Wait() 返回后，给 50ms grace period 让 scanner 把内核 pipe buffer 里最后几行 drain 出来，再强关 fd。

第二层，engine 有两份真相。
core/engine.go 维护一张 interactiveStates map，core/session.go 的 Session 结构体自己也有 busy 字段。
同一个会话的状态被两处分别记录，生命周期还错位—— cleanupInteractiveState 先删 map entry，再异步调 closeAgentSessionWithTimeout。

在这个异步的窗口里，新消息查 map 查不到，走"创建新会话"路径；Session 对象还活着，继续往同一个 chat 推流。两条回复由此产生。

但这还没完，让我好奇的是，究竟为什么会产生两份不同的 truth，为了探究这个问题，我让 agent 帮我梳理了一遍, 于是就有了第三层

第三层，四次 commit 累积出来的裂痕。

  • 3b097a8 初始化：Session 带 busy，合理。
  • ecd0e00 引入 interactiveStates map：为了 engine 层调度，合理。
  • f292073 给 cleanup 加 timeout，改成"先删 map 再异步 close"：为了避免 cleanup 卡主流程，合理。
  • 4089f78 handleMessage 开始读 map 判断忙态：为了队列化并发消息，合理。

每一步都局部正确。拼到一起就散架。

AI 的盲区

这就是我现在越来越警惕的东西：AI 很容易陷入局部正确。

它看当前这个 PR、这个函数、这几行 diff，能给出"正确"的修改。但它缺少一样东西——之前写这段代码的人当时的设计意图。那份意图从来不在代码里，它在作者脑子里，偶尔泄漏到 commit message 或 PR 讨论里，但大部分时候是沉默的。

interactiveStates 这张 map 被引入时，作者心里假设的是"map 存在 ≡ 会话忙"。Session.busy 被设计时，作者心里假设的是"Session 对象在 ≡ 会话活着"。这两个不变量各自成立。没人明确写下来，更没人说过它们之间的关系。

六周后，另一个人（或另一个 AI session）加 cleanup timeout，改成异步 close——从他当时的上下文看，这是合理的小优化。他没理由知道这个改动会让两个不变量第一次脱钩。

人做这件事会怎么做？ 我自己读代码，会下意识推断：为什么这里有两份状态？作者当初是想表达什么？这两份状态的一致性是谁在保证？如果推不出来，我会去翻 git blame、翻 PR 讨论、去群里问一句"这块设计是什么意图"。上下文越长、经验越多，这种"嗅到不对劲"的能力越强。

AI 不会这么做。它默认眼前的代码就是真相，直接基于表面语义推修改。更糟的是 vibe coding 的快反馈环——写、跑、过、提交——在"跑过了"和"提交了"之间，没有留出"我刚才到底动了哪个不变量"的反思空间。

部分解：契约编程

我不觉得这个问题能被完全解决，但至少有一个方向能缓解：把隐式契约显式化，不仅要面向功能编程，更要面向契约编程（Design by Contract）。并且，我们要把契约的受众从“人类”扩展到“AI”。

具体可以落实为三个防线：

1. 架构层：用状态机收敛“唯一事实来源”（Single Source of Truth）

  • 不要让两份独立的数据（如 Map 的存在性与 busy 字段）共同拼凑一个业务事实，这是所有并发灾难的温床。 如果状态耦合是不可避免的约束，就必须在代码结构上强制收敛。例如，暴露一个显式的状态机 Session.GetStatus() -> {Idle, Busy, Cleaning, Closed}。让“清理 Map”和“异步 Close”统一受状态机的流转管辖。一旦检测到状态的非法流转，系统应当选择 Fail-fast（比如直接 Panic 或拒绝服务），也绝不要静默地产生“精神分裂”的脏回复。

2. 认知层：面向 AI 编写“防御性元注释”（Meta-Comments）

  • 在 AI 时代，注释更是写给大模型看的 System Prompt。 AI 拥有极强的代码阅读能力，却毫无工程直觉，它看不见跨文件的生命周期绑定关系。因此，我们需要把隐式契约显式化：在关键变量旁写下类似 // @Invariant: Map 的存在性与 Session 生命周期严格绑定，严禁在 Close 彻底完成前 Delete 的断言。当你写下这句话，就是在给未来接手这块代码的 AI 划定上下文边界，防范它的“局部优化”越过雷池。

3. 编译时契约：用 AST 和静态分析打造“物理隔离”

  • 元注释终究是软约束，AI 也有产生幻觉或被上下文截断的概率。要彻底拦截那些破坏不变量的“局部优化”，我们需要把契约编码为流水线上的“硬约束”。 借助 AST（抽象语法树）分析工具或者自定义 Linter（例如在 Go 中编写自定义的 go/analysis 规则），我们可以将隐式规则转化为可执行的静态检查。比如，针对这个并发 Bug，我们可以写一条专门的 Lint 规则：在全局代码中拦截所有针对 interactiveStates 的 delete 操作，分析其 AST 上下文，如果该操作没有被包在一个校验了 Session.GetStatus() == StatusClosed 的条件分支内，直接在本地 Commit 或 CI 阶段报错拦截。

What else?

真正的破局，应该发生在 Agent 框架层

前面提到的状态机、元注释和静态分析，本质上都是我们在现有的 AI 交互模式下，被动做出的防御。但如果我们把目光放长远，防范局部正确陷阱的终极责任，不应该全压在开发者身上，而应该由 AI 的底层工程框架（Harness Engineering）来承担。

它不应仅仅是文档检索器，而是要充分榨取 LLM 的高并发能力，在修改代码的同时并行触发多维任务：

  • 向上自动追溯 Git 历史、ADR 及 PR 讨论以还原“沉默的意图”（若有断层则主动向人类发起询问）
  • 横向拉起 Agent 对所有关联模块进行 “Map-Reduce 式” 的并行冲突扫描，从而有效对修改进行评估。
  • 纵向则可以拉起多个 Agent 同步生成方案，分别给不同的提示词和温度，生成完毕后调用审查 Agent 进行评估，避免大模型过早收敛到某条思路上。

收尾

局部正确的陷阱，是 AI 辅助编程从“效率玩具”迈向“工业级可靠性”必须跨越的鸿沟。今天，我们用显式契约和状态机来保卫系统；但明天，真正能降服大模型这种“局部优化怪兽”的，不是更聪明的模型本身，而是更具架构深度的 Agent 框架。

只有当我们的基建不再盲目追求“快反馈”，而是学会了“慢思考”、懂得了精简上下文、并敢于向人类逼问意图时，人机协作的编程范式，才算迎来了真正的破局。

第 2 步：状态 A（interactiveStates map）

Commit: ecd0e00 (2026-02-28 22:05) — feat(claudecode): refactor agent into persistent session model …（同一天晚些）

为了支持持久化 claudecode session + 权限状态，在 core/engine.go 里引入 interactiveStates map[string]*interactiveState 和 interactiveMu。从这一刻开始，「turn 在跑」有了两份独立的记账。

在这个 commit 里 cleanup 还是原子的（mu 锁住整个 Close + delete），所以两份状态的视图虽然分裂但在时间维度上没有窗口差：

Go

同期的 cmdStop 也只读 map（和今天一样）。这两份真相此时已经存在，但由于 cleanup 在 mu 下串行化，读者要么看到 state-still-present、要么看到 state-gone，没有「map 空但 agent 还在收尾」的中间态。

第 3 步：把「原子 cleanup」拆成「先 delete 后 async-close」

Commit: f292073 (2026-03-04) — feat: add timeout handling for agent session closure

这个 commit 改了 cleanupInteractiveState：先 delete 再释放 interactiveMu，然后把 Close 丢到新 goroutine 里用 10s timeout 等（后面涨到 130s）：

Go

这是两份真相第一次在时间维度分开。map 项消失、但 session lock 和 Close 仍在生效。原来被 interactiveMu 吸收掉的等待时间现在泄漏成了一个「状态已删但工作仍在」的窗口。

第 4 步：让 handleMessage 的 busy 分支开始读 map

Commit: 4089f78 (2026-03-16) — feat: queue messages when agent is busy instead of discarding them

这之前，handleMessage 走到 busy 分支是无条件回 MsgPreviousProcessing——和 cmdStop 一样只根据一个信号决定回复，两者不会矛盾。

这个 commit 引入了 queueMessageForBusySession：TryLock 失败后先去读 interactiveStates 决定「排队还是回 busy」。这就让 handleMessage 的回复策略变成了 session.busy + interactiveStates 联合判定，而 cmdStop 仍然只看后者。

Go

从这一刻起，同一份瞬态「map 空 + session lock 持有」下：

  • handleMessage 走 queue miss → MsgPreviousProcessing（因为 TryLock 失败且 queue 说没 state）
  • cmdStop 走 stopInteractiveSession miss → MsgNoExecution
