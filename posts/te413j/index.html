血泪教训：Agent 提 PR 太快把号干封了，聊聊我的 MCP + 沙盒防封折腾记录

写在前面

作为一个折腾爱好者，最近在弄让 AI Agent 自动跑项目、提 PR 的工作流，结果踩了个大坑——专门给 Agent 注册的 GitHub 小号，因为一开始没做限制，被它用裸 gh cli 短时间内狂刷 PR 和 Comment，直接被 GitHub 的防滥用风控当场封禁（Suspended）。

痛定思痛，我寻思给它切到 MCP（模型上下文协议）总行了吧？结果发现光给它配个 MCP 工具，根本防不住大模型偶尔的抽风。今天纯分享一下我最后是怎么用 MCP 配合网络沙盒，把这套工作流给彻底“管”起来的。里面包含了完整的架构思路，还有 MCP Proxy 和 Squid 代理的 Docker 实战代码，希望能帮大家少废几个小号。

为什么切了 MCP 还是会出问题？

被封号之后，我觉得直接用终端命令太不可控了，于是换上了现成的 GitHub MCP Server。毕竟 MCP 底层封装了 Octokit，遇到 API 限流（429 报错）会自动触发退避等待，很安全。

但实操后发现一个很让人抓狂的现象：Agent 经常放着配好的 MCP 不用，非要自己手搓 curl 去直接调 API。

其实想想也正常：

  • 大模型的“肌肉记忆”：在它吃过的海量训练语料里，遇到 GitHub API，第一反应绝对是写一段 curl 代码，而不是去读你刚塞给它的 MCP 工具说明书。
  • Bash 环境太方便了：只要你给了它执行终端命令的权限，它就会觉得直接敲命令最省事。
  • curl 没有限流保护：一旦它绕开 MCP 自己写脚本报错了，为了修 Bug 就会开始疯狂重试，一秒钟砸过去十几个 curl 请求，GitHub 分分钟又要拿你当水军封号。

怎么治？直接“物理断网”

在 System Prompt 里写“严禁使用 curl”基本没用，大模型一上头全忘了。综合踩坑经验后我发现，最有效的办法就是在基础设施层直接切断它的违规路径。

既然如此，我把整个架构拆成了三块，放进了 Docker 里进行严格的网络隔离：

  • 被关禁闭的 Agent 容器：给它分配一个内部的局域网（Internal Network），完全不给公网访问权限。Agent 在里面怎么折腾都行，但只要它敢在终端里敲 curl https://api.github.com，立马报错 Network is unreachable。
  • 特权管家 MCP Server 容器：这里单独跑一个 MCP Server 代替它操作 GitHub。这个容器允许连外网，并且配置了 GitHub 的 Fine-grained PAT。
  • 出网网关（Egress Proxy）容器：给 Agent 开一个只允许拉包查资料，严禁触碰 GitHub API 的出网代理。

它们怎么通信？
只要让 Agent 通过 HTTP SSE 去呼叫内网里的 MCP Server 就行了。Agent 撞了几次南墙发现网络不通，自然就乖乖地按 JSON 规范去调 MCP 接口了。这样一来流量全走 MCP，限流和防刷屏的机制终于生效了。

官方 MCP 的坑

既然已经决定了架构，首先要把 MCP Server 作为一个独立的服务跑起来。但如果你直接用 GitHub 官方的 MCP 库（@modelcontextprotocol/server-github），会踩到一个大坑：它默认只支持 Stdio（标准输入输出），不支持作为 HTTP 服务被其他容器调用。

预想方案：把“上网”也变成 MCP 工具
后面我打算连代理都不给了，让 Agent 彻底断网。然后在沙盒外单独部署一个专门用来上网的 Web-Surfing MCP Server，提供 search_web 和 fetch_url 工具。Agent 如果遇到没见过的报错，只能调用这个工具去搜。
理论上这样做有个极大的好处：MCP 服务端可以把抓回来的网页，用 BeautifulSoup 洗一遍，变成干净的 Markdown 再喂给大模型。不仅 Token 消耗更少，而且能绝对杜绝它自己瞎发网络请求的风险。目前这个方案还在构思中，等后面有空了折腾一下试试。

效果验证

启动服务（docker-compose up -d）后，你可以进入 Agent 容器内部，亲自模拟一下大模型的行为来验证防御效果：

  1. 合法出网测试（拉取代码依赖）：
    在容器内执行 curl -I https://registry.npmjs.org。返回 HTTP/2 200，Squid 代理正常放行，Agent 可以拉包。
  2. 违规操作测试（企图绕开 MCP 调 GitHub API）：
    在容器内执行 curl -I https://api.github.com/user。瞬间返回 HTTP/1.1 403 Forbidden。Squid 直接把请求拦截并抛回异常，GitHub 的风控系统根本不会知道有这回事。
  3. 合规操作测试（乖乖使用 MCP）：
    Agent 撞墙报错后，向配置好的 http://mcp-gateway:8080/sse 发送 JSON 工具调用请求。请求通过内部局域网瞬间送达，MCP 代为执行合法的频率控制与 API 交互。

总结

其实踩完这个坑就明白了一件事：千万别把 Agent 当成一个懂规矩的同事。只要给了它通用终端的权限，它其实就是个随时可能写出死循环的脚本小子。

不要试图用 Prompt 去劝它向善，直接用网络隔离加上应用层代理把危险操作的口子收紧，并强制通过 MCP 协议进行外部交互，才是最省心的路子。
